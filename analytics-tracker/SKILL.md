# Analytics Tracker

扫描项目 → 生成追踪模块 → 修改现有文件 → 指导平台配置。

## 阶段 1：询问配置

开始前，向用户确认以下信息：

1. **主平台**：GA4 / Umami / Plausible（三选一）
2. **追加平台**（可选）：是否同时上报到其他平台？
3. **项目目录**：需要扫描哪个目录？（默认当前工作目录）

示例询问：
> "请告诉我：① 你的主要分析平台是 GA4、Umami 还是 Plausible？② 是否需要同时上报到其他平台？③ 项目根目录路径是？（直接回车使用当前目录）"

---

## 阶段 2：扫描项目

加载 `references/event-patterns.md`，按其中的检测模式扫描项目文件。

**扫描范围**：`.js` `.ts` `.jsx` `.tsx` `.vue` `.svelte`
**跳过**：`node_modules/`、`*.test.*`、`*.spec.*`、`*.config.*`、`dist/`、`build/`

按 `event-patterns.md` 中的优先级规则，将候选位置分为 P1/P2/P3 三级，
使用 `event-patterns.md` 中定义的**候选清单输出格式**展示结果。

**边界情况**：
- 若扫描后 P1 候选为零 → 提示用户"未找到关键业务事件，请检查项目目录是否正确，或手动指定需要埋点的文件"
- 若扫描文件超过 50 个 → 只展示 P1 和 P2，P3 候选数量过多时合并为"共 N 个导航/通用点击事件"

**等待用户确认**：
- 是否删除某些条目？
- 是否手动添加遗漏位置？
- P3 中是否有需要保留的？

用户确认后，进入阶段 3。

---

## 阶段 3：生成追踪模块

加载 `references/analytics-template.md`。

**检查是否已有追踪模块**：先搜索项目中是否存在 `analytics.ts`、`analytics.js`、`lib/analytics.*`、`utils/analytics.*`。
- 若已存在 → 展示现有内容，询问"是更新现有文件还是创建新文件"；更新时只在区域 3 追加新事件函数，不覆盖已有内容
- 若不存在 → 按模板创建新文件

**检测项目语言**：
- 有 `tsconfig.json` → 生成 `analytics.ts`
- 否则 → 生成 `analytics.js`（移除类型注解）

按模板结构生成文件：
- **区域 1**：按用户在阶段 1 的选择填入 `primary` 和 `additional`
- **区域 2**：直接使用模板中的核心分发函数（样板，不修改）
- **区域 3**：只生成用户在阶段 2 确认的事件（P3 默认不生成）

**文件放置**：按以下优先级选择：`src/analytics.*` → `lib/analytics.*` → 项目根目录。若以上路径均无对应父目录，询问用户确认。

展示生成的文件内容，等待用户确认后写入磁盘。

---

## 阶段 4：修改现有文件

对阶段 2 确认的每个候选位置，生成最小化 diff：

```diff
+ import { track } from '@/analytics';  // 或相对路径

  <button onClick={() => {
+   track.signupSubmit('email');
    handleSignup();
  }}>注册</button>
```

**操作方式**：
- 逐文件展示 diff 预览
- 提供选项：确认此文件 / 跳过 / 全部确认

每处改动只做两件事：添加 import（如已有则不重复添加）+ 插入追踪调用。不修改原有逻辑。

**import 路径规则**：
- 有 `tsconfig.json` 且其 `paths` 字段配置了 `@/*` 或类似别名 → 使用别名（如 `@/analytics`）
- Vue 项目有 `vite.config.*` 且其中配置了 `resolve.alias` → 使用对应别名
- 否则 → 使用相对路径（如 `../analytics` 或 `../../analytics`，根据文件相对位置计算）

---

## 阶段 5：平台配置指南

根据用户在阶段 1 选择的平台，加载对应的 references 文件：

- GA4 → 加载 `references/ga4.md`
- Umami → 加载 `references/umami.md`
- Plausible → 加载 `references/plausible.md`

如有追加平台，按顺序逐个完成配置。

按 references 文件中的三步结构指导用户：
1. **基础安装**（获取 ID → 安装脚本）
2. **自定义事件配置**（后台操作）
3. **验证**（触发操作 → 在平台后台确认）

**验证节点**：每个平台完成第三步后，明确提示：
> "请在浏览器中触发一次 [事件名] 操作，然后在 [平台名] 后台确认事件出现。完成后告诉我，我们继续下一步。"

---

## 约束

- **扫描时不修改任何文件**，只读取和分析
- **代码改动最小化**：每处只加 import + 追踪调用，不重构原有代码
- **两个用户确认门**必须经过：阶段 2 候选清单确认、阶段 4 每文件 diff 确认
- **P3 事件默认不生成**，除非用户明确选择
- **不重复添加 import**：若文件已有 `from '@/analytics'` 则只添加新的 `track.*` 调用
